# ドキュメント提案ガイドライン

機能開発後に、ユーザー向けドキュメントの更新が必要かを判断し提案するためのガイドライン。

---

## 提案タイミング

以下のタイミングでドキュメント更新の提案を行う:

1. **機能実装の完了後**（コミット前後）
2. **PR作成時**
3. **ユーザーから `/write-docs` コマンドで明示的に依頼された時**

---

## 提案判断マトリクス

| 変更の種類                 | 提案するコンテンツ                                              | 優先度 |
| -------------------------- | --------------------------------------------------------------- | ------ |
| 新機能追加                 | `features/` ページ新規 or 更新 + ブログ記事                     | 高     |
| 既存機能の大幅変更         | `features/` ページ更新                                          | 高     |
| UI/UX変更                  | `getting-started/ui-overview` 更新、該当 `features/` ページ更新 | 中     |
| バグ修正（ユーザー影響大） | `troubleshooting/` 追記                                         | 中     |
| リリース                   | リリースノート作成                                              | 高     |
| 新しいショートカット追加   | `features/shortcuts.mdx` 更新                                   | 高     |
| 設定項目追加               | `account/` 配下の更新                                           | 中     |
| パフォーマンス改善         | ブログ記事候補                                                  | 低     |
| 使い方パターンの変化       | `guides/` 配下の更新 or 新規                                    | 中     |
| 内部リファクタリング       | 提案なし                                                        | -      |
| テスト追加のみ             | 提案なし                                                        | -      |
| 依存関係更新のみ           | 提案なし                                                        | -      |

---

## 提案しない場合

以下の変更ではドキュメント提案を行わない:

- 内部実装の変更のみ（ユーザーに見える変化がない）
- テスト・CI/CD の変更
- 開発者向けツールの変更
- コードフォーマット・リント修正

---

## 提案フォーマット

提案時は以下のフォーマットで簡潔に提示:

```markdown
---

### ドキュメント更新の提案

今回の変更に伴い、以下のユーザー向けドキュメントの更新を提案します:

| 対象 | 操作 | 理由 |
|------|------|------|
| `content/docs/features/plans.mdx` | 更新 | 繰り返し予定の新オプションを追記 |
| `content/blog/recurring-plans-tips.mdx` | 新規 | ユーザー向けに活用方法を紹介 |

`/write-docs` コマンドで執筆を開始できます。

---
```

## ブログ記事のアイデア例

機能開発後に書けるブログ記事のパターン:

### ユーザー視点の活用Tips

- 「タイムボクシングで生産性を上げる5つのコツ」
- 「Dayoptで始める週次レビュー習慣」
- 「タグの親子階層を活用した時間分析」

### 機能紹介・アップデート

- 「新機能: ドラッグ&ドロップで予定を瞬時に調整」
- 「キーボードショートカットで2倍速く操作する方法」
- 「カレンダービューの使い分けガイド」

### 開発の裏側（技術ブログ）

- 「楽観的更新で体感速度を200ms改善した話」
- 「リアルタイム同期の設計と実装」

## navigation.ts の更新判断

| ドキュメント操作 | navigation.ts の対応              |
| ---------------- | --------------------------------- |
| 既存ページの更新 | 変更不要                          |
| 新規ページの追加 | エントリを追加                    |
| ページの削除     | エントリを削除                    |
| ページ名の変更   | `title` を更新                    |
| セクションの追加 | 新しい `NavigationSection` を追加 |
