# Layer 2: Pattern Detection

統計ベースの傾向分析。LLMを使わず、集計データからパターンや分布を可視化する。

---

## 位置づけ

```
Layer 1 (自動計算) — KPI数値の算出
  ↓ データを提供
Layer 2 (パターン検出) ← 今ここ — 集計データからパターンや分布を可視化
  ↓ 構造化データを提供
Layer 3 (LLMインサイト) — 自然言語の気づき生成
```

- **対象ユーザー**: 全ユーザー（Free含む）
- **AI依存**: なし（純粋な統計処理）
- **データソース**: PostgreSQL RPC関数 → tRPC → TanStack Query → Recharts
- **現在の配置**: 主に Progress タブ（累積データ表示）

> **Note**: Layer とタブは直交する概念。タブは「期間振り返り（Review）/ 累積（Progress）」というデータの性質で分類し、Layer は計算の複雑度で分類する。Layer 2 のチャートが Review タブに移動する、またはその逆もありうる。

## データフロー

```
PostgreSQL RPC関数
  → entries統計ルーター (src/features/entry/server/statistics.ts)
    → TanStack Query (並列取得・自動キャッシュ)
      → Recharts チャートコンポーネント
```

### 日付範囲の計算

チャートは `useStatsFilterStore` の `currentDate` と `granularity` から日付範囲を算出:

| ユーティリティ | 用途 | 使用チャート |
|-------------|------|------------|
| `computeStatsDateRange()` | 粒度→開始/終了日 | DayOfWeek, HourlyDistribution |
| `computeMonthCount()` | 粒度→取得月数 | MonthlyTrend |

**粒度と月数の対応:**
- `day` → 1ヶ月 / `week` → 3ヶ月 / `month` → 12ヶ月 / `year` → 全期間

YearlyHeatmap は独自の年ナビゲーション（`useState`）で年を指定。

## 追加候補チャート

MetricsCatalog.docs.mdx で Layer 2 向きと判定されたもの:

| チャート | データソース | 表現方式 | 優先度 |
|---------|------------|---------|--------|
| **充実度×時間帯** | `getEnergyMap` | 2Dヒートマップ（hour × dow） | ★★★ |
| **見積もりバイアス推移** | `getEstimationAccuracy` 時系列化 | 折れ線グラフ（月次） | ★★☆ |
| **曜日×充実度クロス** | `getEnergyMap` | 棒グラフ + 充実度オーバーレイ | ★★☆ |
| **タグ間遷移パターン** | 新規RPC | Sankey or マトリクス | ★☆☆ |
| **ルーティン遵守率** | 新規RPC（繰り返し検出） | リスト + 達成率 | ★☆☆ |

### 充実度×時間帯ヒートマップ（推奨次期実装）

既存の `getEnergyMap` endpoint が `{ hour, dow, avg_fulfillment, total_minutes }` を返すので、
追加のDB関数なしで実装可能。chronotypeのピーク時間帯をオーバーレイすると Layer 1 の
Peak Utilization と連動した体験になる。

## Layer 1 からの接続点

Layer 1 の KPI メトリクスが提供するデータを、Layer 2 でパターンとして可視化:

| Layer 1 メトリクス | Layer 2 での展開 |
|------------------|----------------|
| Plan Rate | 計画習慣の週次/月次トレンドライン |
| Estimation Accuracy | タグ別の見積もりバイアス推移 |
| Energy Map (Peak Util) | 充実度×時間帯の2Dヒートマップ |
| Context Switches | 曜日パターンによる集中日/分散日の検出 |

---

## Layer 3 への接続点

Layer 2 のチャートデータを構造化テキストとして LLM に渡す想定:

- 月次トレンドの方向性（上昇/下降/横ばい）
- 最も生産的な曜日・時間帯
- 見積もり精度の改善/悪化パターン
- 異常値（突出した日/週）の検出
