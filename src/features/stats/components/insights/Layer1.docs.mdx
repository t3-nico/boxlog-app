# Layer 1: Auto-Computed Metrics

LLMを使わない純粋な集計ロジック。エントリデータから自動計算されるKPI数値を提供する。

---

## 位置づけ

```
Layer 1 (自動計算) ← 今ここ
  ↓ データを提供
Layer 2 (パターン検出) — 統計ベースの傾向分析
  ↓ 構造化データを提供
Layer 3 (LLMインサイト) — 自然言語の気づき生成
```

- **対象ユーザー**: 全ユーザー（Free含む）
- **AI依存**: なし
- **データソース**: PostgreSQL RPC関数 → tRPC → TanStack Query
- **表示場所**: Stats > Insights タブ

---

## メトリクス一覧（8カード）

| メトリクス | 計算式 | DB関数 | 意味 |
|-----------|--------|--------|------|
| **Total Time** | `SUM(duration)` | `getCumulativeTime`（既存） | 期間内の記録時間合計。最も基本的な活動量 |
| **Avg Fulfillment** | `AVG(fulfillment_score)` | 新規RPC or client計算 | 充実度の全体平均。Dayopt独自の「時間の質」指標 |
| **Plan Rate** | `planned / total` | `get_plan_rate` | 計画して行動できた割合。高いほど意図的な時間の使い方 |
| **Streak** | 連続アクティブ日数 | `getStreak`（既存） | 継続日数。モチベーション維持に直結 |
| **Estimation Accuracy** | `avg(\|actual - planned\|)` | `get_estimation_accuracy` | 見積もりと実績のズレ。小さいほど自己理解が正確 |
| **Peak Utilization** | `peak_minutes / peak_available` | `get_energy_map` + client計算 | ピーク時間帯をどれだけ活用しているか |
| **Context Switches** | `tag変化回数 / 日数` | `get_context_switches` | 1日あたりのタスク切替。多すぎると集中力が分散 |
| **Blank Rate** | `1 - scheduled / available` | `get_blank_rate` | 空白時間の割合。余白は悪ではなく、意図的かどうかが重要 |

### カード配置

```
Row 1: Total Time | Avg Fulfillment | Plan Rate    | Streak
Row 2: Est. Acc.  | Peak Util.      | Ctx Switches | Blank Rate
```

- **グリッド**: `grid-cols-2 sm:grid-cols-4`（モバイル2列、SM以上4列）
- Row 1 = 基本指標（量・質・計画・継続）、Row 2 = 分析指標（精度・ピーク・集中・余白）

## アーキテクチャ

### データフロー

```
PostgreSQL RPC関数
  → entries統計ルーター (src/features/entry/server/statistics.ts)
    → TanStack Query (並列取得・自動キャッシュ)
      → 純粋計算関数 (src/features/stats/lib/metrics.ts)
        → UIコンポーネント
```

### ファイル構成

```
src/features/stats/
├── types/
│   └── metrics.types.ts        型定義（PlanRateData等）
├── lib/
│   ├── metrics.ts              純粋計算関数（ピーク活用率、フォーマット、トレンド）
│   └── metrics.test.ts         ユニットテスト（20テスト）
└── components/
    ├── metrics/
    │   └── MetricCard.tsx       KPIカード（値 + トレンド + 説明）
    └── insights/
        ├── StatsMetricsGrid.tsx 5カードのグリッド（tRPC接続）
        ├── EstimationAccuracyChart.tsx  タグ別予定vs実績（Pure + Connected分離）
        └── InsightsView.tsx     Insightsタブのレイアウト
```

### Feature Boundary

- DB集約（RPC呼び出し）は `entry` feature が所有（既存パターンを踏襲）
- 表示ロジック・UIは `stats` feature が所有
- chronotypeデータは `useCalendarSettingsStore`（共有store）経由

## コンポーネント

### MetricCard

単一のKPI数値を表示するカード。

| Prop | 型 | 説明 |
|------|----|------|
| `label` | `string` | メトリクス名（"Plan Rate"等） |
| `value` | `string` | フォーマット済みの値（"72%", "12m"等） |
| `description` | `string` | 補足説明 |
| `trend` | `MetricTrend?` | 前期間比（up/down/flat + delta） |
| `isLoading` | `boolean?` | スケルトン表示 |

### EstimationAccuracyChart

タグ別の予定時間 vs 実績時間を横棒グラフで比較。

- `EstimationAccuracyChartPure`: props駆動（Storybook用）
- `EstimationAccuracyChart`: tRPC接続版（アプリ用）

### StatsMetricsGrid

8つのMetricCardを2行×4列グリッドに配置。各メトリクスのtRPCクエリを並列実行。

### InsightsView

Insightsタブのルートコンポーネント。Phase 2でパターン検出コンポーネントが追加される。

## 純粋計算関数

### `calculatePeakUtilization(energyMap, peakZones, daysInRange)`

エネルギーマップデータとchronotypeピークゾーンからピーク活用率を算出。

### `formatMetricValue(value, type)`

数値を表示用文字列に変換。`percentage`→"72%", `minutes`→"1h 30m", `count`→"3.2"

### `getMetricTrend(current, previous)`

前期間比のトレンドを計算。5%未満の変化は`flat`。

## DB関数

### get_plan_rate(user_id, start_date, end_date)

```sql
RETURNS JSON -- { totalEntries, plannedEntries, planRate }
```

### get_estimation_accuracy(user_id, start_date, end_date)

```sql
RETURNS TABLE (tag_id, tag_name, tag_color, avg_planned_minutes,
               avg_actual_minutes, avg_deviation_minutes, entry_count)
-- HAVING count >= 3（最低3サンプル）
```

### get_context_switches(user_id, start_date, end_date)

```sql
RETURNS JSON -- { totalSwitches, avgPerDay }
-- LAG()ウィンドウ関数で連続エントリのタグセット変化を検出
```

### get_blank_rate(user_id, start_date, end_date, wake_hour, sleep_hour)

```sql
RETURNS JSON -- { availableMinutes, scheduledMinutes, blankMinutes, blankRate }
-- 活動可能時間帯のみを対象（デフォルト7:00-23:00）
```

### get_energy_map(user_id, start_date, end_date) — 既存

```sql
RETURNS TABLE (hour, dow, avg_fulfillment, total_minutes, entry_count)
-- ピーク活用率の算出にクライアント側で利用
```

## Layer 2 への接続点

Layer 1のデータはLayer 2のパターン検出の入力になる:

- **Plan Rate** の推移 → 計画習慣の改善/悪化トレンド
- **Estimation Accuracy** のタグ別データ → 見積もりバイアスの傾向検出
- **Energy Map** → 充実度×時間帯のクロス集計、ルーティン検出
- **Context Switches** の曜日パターン → 集中日/分散日の検出
